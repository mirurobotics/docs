       # v0.9.0

       *May 5, 2026*

       Miru Agent `v0.9.0` introduces provisioning tokens as the standard programmatic device-provisioning mechanism, replacing the legacy API-key flow.

       To learn more, visit the [provisioning tokens](/docs/learn/devices/provision/provisioning-tokens) documentation.

       ## Breaking changes

       - API-key based provisioning is no longer supported; use [provisioning tokens](/docs/learn/devices/provision/provisioning-tokens) instead. Devices previously activated via API keys must be [reprovisioned](/docs/learn/devices/provision/reprovision) after upgrading
       - The agent installer is now invoked as `provision` (and `reprovision` for re-enrollment); device name is now a required field on enrollment
       - Backend URL and MQTT host are now restricted to `mirurobotics.com` and its subdomains (loopback also permitted); MQTT requires TLS except on loopback. If on-device settings are tampered with, the agent falls back to production defaults instead of failing silently
       - Existing pre-`v0.9.0` agent install scripts will refuse to install `v0.9.0` or later. Follow the updated [install instructions](/docs/developers/agent/install)

       ## Features

       - Added `provision` and `reprovision` subcommands for enrolling and re-enrolling devices using provisioning tokens
       - On agent version change, per-version on-device state is reset and rebootstrapped from the backend so the device stays consistent across upgrades; the device's auth keypair is preserved so the device retains its identity

       ## Fixes

       - Fixed a Debian `postinst` failure (`useradd: group miru exists`) on systems with `USERGROUPS_ENAB=yes`; added a `tmpfiles.d` entry so `/srv/miru` is recreated on boot
       - Resolved a `rustls-webpki` security advisory by routing MQTT TLS through `native-tls`
       - Fixed a `usize` underflow in cache pruning that, in release builds, could wipe all valid cache entries after invalid entries were removed

       ## Improvements

       - `settings.log_level` now takes effect after the agent reloads its configuration, without restart
