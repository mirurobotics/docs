---
title: 'File System Permissions'
---

The Miru Agent runs as the unprivileged `miru` system user and group. When the agent deploys configuration instances, it writes files to absolute paths on the filesystem. If those paths fall outside the agent's default writable directories, you need to grant the agent access.

This page covers the default permissions model, how to check what the agent can access, and how to grant access to custom paths.

## Defaults

During installation, the agent package creates a `miru` system user and group with no login shell. The agent process runs as `miru:miru` under systemd with `ProtectSystem=strict`, which makes the entire filesystem read-only except for explicitly allowed paths.

The agent has write access to these directories by default:

| Path | Purpose | Permissions |
|------|---------|-------------|
| `/var/lib/miru` | Agent state, credentials, device identity | `0755`, `miru:miru` |
| `/var/log/miru` | Log files (hourly rotation) | `0755`, `miru:miru` |
| `/srv/miru` | Deployed configuration instance files | `0755`, `miru:miru` |

The agent also uses a Unix domain socket at `/run/miru/miru.sock` (mode `0660`, group `miru`) for its local [Device API](/docs/developers/device-api/overview). Applications that need to communicate with the agent should be added to the `miru` group.

## Checking access

### Test if a directory is accessible

To test whether the agent has read, write, and execute access to a directory:

```bash
sudo -u miru test -r /path/to/dir \
  && sudo -u miru test -w /path/to/dir \
  && sudo -u miru test -x /path/to/dir \
  && echo "accessible" || echo "not accessible"
```

### Test if a file is accessible

To test whether the agent has read and write access to a file:

```bash
sudo -u miru test -r /path/to/file \
  && sudo -u miru test -w /path/to/file \
  && echo "accessible" || echo "not accessible"
```

### List accessible contents

To see which files and directories the agent can read within a path:

```bash
sudo -u miru find /path/to/dir -maxdepth 1 2>/dev/null
```

If ACLs are in use, you can also inspect the full permission set:

```bash
getfacl /path/to/dir
```

## Granting access

There are two layers to consider when granting the agent access to a custom path:

1. **Unix permissions** — the `miru` user or group must have the required permissions on the target path
2. **systemd ReadWritePaths** — the path must be included in the agent's `ReadWritePaths` directive, since `ProtectSystem=strict` makes all other paths read-only

Both layers must allow access. Unix permissions alone are not sufficient if the path is outside `ReadWritePaths`.

### Standard Unix permissions

The simplest approach is to give the `miru` user or group ownership of the target path.

#### Grant access to a directory

**Option A — Transfer ownership** (when the directory is exclusively for the agent):

```bash
sudo chown -R miru:miru /path/to/dir
sudo chmod -R u+rwx /path/to/dir
```

**Option B — Add group access** (when the directory is shared with other users):

```bash
sudo chgrp -R miru /path/to/dir
sudo chmod -R g+rwx /path/to/dir
```

#### Grant access to a file

**Option A — Transfer ownership**:

```bash
sudo chown miru:miru /path/to/file
sudo chmod u+rw /path/to/file
```

**Option B — Add group access**:

```bash
sudo chgrp miru /path/to/file
sudo chmod g+rw /path/to/file
```

### Update systemd ReadWritePaths

After setting Unix permissions, add the path to the agent's `ReadWritePaths` so systemd allows writes through `ProtectSystem=strict`.

Create a systemd drop-in override:

```bash
sudo systemctl edit miru.service
```

This opens an editor. Add the following content:

```ini
[Service]
ReadWritePaths=/path/to/dir
```

Save and exit, then reload and restart the agent:

```bash
sudo systemctl daemon-reload
sudo systemctl restart miru.service
```

You can verify the override is active:

```bash
systemctl cat miru.service
```

The drop-in section should appear at the end of the output, showing your custom `ReadWritePaths`.

<Note>
  `ReadWritePaths` entries in drop-in files are additive — they extend the paths defined in the base unit file rather than replacing them. The agent's default paths (`/var/lib/miru`, `/var/log/miru`, `/srv/miru`) remain writable.
</Note>

### ACLs for fine-grained control

When you cannot change ownership of a path — for example, a directory shared between multiple services — use POSIX Access Control Lists (ACLs) for fine-grained permission control.

<Note>
  ACLs require the `acl` package. Install it with `sudo apt install acl` if `setfacl` is not available.
</Note>

#### Grant access to a directory

Set ACLs on existing contents and a default ACL for new files created within the directory:

```bash
# Grant access to existing files and subdirectories
sudo setfacl -R -m u:miru:rwx /path/to/dir

# Set default ACL so new files inherit miru access
sudo setfacl -R -d -m u:miru:rwx /path/to/dir
```

#### Grant access to a file

```bash
sudo setfacl -m u:miru:rw /path/to/file
```

#### View current ACLs

```bash
getfacl /path/to/dir
```

The output shows both standard Unix permissions and any ACL entries. Look for lines starting with `user:miru:` to confirm the agent's access.

<Note>
  Even with ACLs set, you still need to add the path to `ReadWritePaths` if `ProtectSystem=strict` is active. ACLs control who can access a path, while `ReadWritePaths` controls whether systemd allows the agent process to reach it.
</Note>
