Create a release along with its associated git commit, config schemas, and optional [upload rules](/data-uploads/releasing-upload-rules) in Miru. 

The git commit, schemas, and upload rules associated with the release are all created idempotently. If the git commit does not yet exist in Miru, it is created. If it already exists, it is not recreated and no error is returned.

Similarly, if the provided schemas do not yet exist in Miru, they are created. If they already exist, they are not recreated and no error is returned.

Git commits are identified by their SHA while schemas are identified by their content. Schemas with equivalent content are considered identical in Miru (even if the comments, spacing, or other formatting is different). Upload rules work the same way: a rule file identical to an existing rule in the same [upload collection](/data-uploads/upload-collections) attaches the existing rule to the release rather than creating a duplicate.

For a more detailed guide on creating releases, visit the [create releases](/cfg-mgmt/releasing-config-schemas) documentation; for including upload rules, see [releasing upload rules](/data-uploads/releasing-upload-rules).

### Requirements

- The release's config schemas must be committed to a local Git repository
- The schemas must be annotated with their config types (see [annotations](#schema-annotations) below)
- The Git repository must have a remote URL (GitHub, GitLab, Bitbucket, etc.) configured
- The release must include at least one config schema — upload rules are optional and, when provided, must also be committed to the Git repository
