The Miru dashboard is the simplest way to provision devices, ideal for getting started or managing individual devices.

For automated deployments or provisioning devices at scale, consider using [API Keys](#api-keys) instead.

## API Keys

To programmatically provision devices, Miru offers API keys.

Unlike the dashboard installation, which requires manual interaction with the web interface for each device, provisioning devices via API keys allows you to:

- **Provision multiple devices** at scale without manual intervention
- **Integrate with existing infrastructure** management tools and workflows
- **Use in headless environments** where a web browser isn't available

### Create the API Key

To create an API key, follow the instructions for [creating an API key](/docs/admin/apikeys#create-an-api-key) and select the `devices:provision` scope.

### Provision the Device

On the device you want to install the Miru Agent, set the `MIRU_API_KEY` environment variable to your API key.

```bash
export MIRU_API_KEY=<your-api-key>
```

Execute the [provisioning script](https://github.com/mirurobotics/agent/blob/main/scripts/install/provision.sh) to create and activate a device using your API key.

```bash
curl -fsSL https://raw.githubusercontent.com/mirurobotics/agent/main/scripts/install/provision.sh | sh -s -- \
  --device-name=$HOSTNAME \
  --allow-reactivation=true
```

The provision script performs the following actions:

1. **Creates the device** - creates a new device in the Miru dashboard if a device with the provided name doesn't already exist
1. **Configures the apt repository** - adds the Miru apt source list and signing key (`/usr/share/keyrings/miru-agent.gpg`) if not already present
1. **Installs the agent** - installs `miru-agent` via `apt-get install`
1. **Installs the service** - installs the agent as a `systemd` service
1. **Activates the device** - registers the agent with Miru servers using the activation token

The provision script supports the following parameters to tune your installation:

<ParamField path="--device-name" type="string">
  The name of the device in the Miru dashboard.

  Default: `$HOSTNAME`
</ParamField>

<ParamField path="--allow-reactivation" type="boolean">
  Allows reactivation of existing devices.

  Without this flag, only `inactive` devices can be activated. With this flag, devices that have been previously activated (status `online`, `offline`, etc.) may be reactivated.

  Default: `false`
</ParamField>

<ParamField path="--version" type="string">
  The agent version to install (e.g., `v0.6.0`).

  Defaults to the latest version available in the apt repository. To list all available versions on a configured machine run `apt-cache madison miru-agent`. The full release list is also browsable on [GitHub Releases](https://github.com/mirurobotics/agent/releases).
</ParamField>

<ParamField path="--from-pkg" type="string">
  The file path to the Miru Agent Debian package to use for installation. Useful for installing the agent in poor connectivity scenarios.
</ParamField>

## Fleet provisioning

The [API Keys](#api-keys) flow above is the simplest way to provision a device — it's well suited to individual devices, evaluation, and getting started. For a production fleet, where you're provisioning hundreds of devices and want pinned versions, audit trails, and no live network dependency on third-party hosts at provision time, we recommend one of the two patterns below.

This is the same separation Docker draws between its convenience installation script (intended for development and testing) and the manual apt setup (intended for production). The Miru `provision.sh` script is a convenience wrapper; the patterns below decompose it into the underlying steps that production tooling can run directly.

### Vendor the install script

If you provision devices through orchestration tooling — Ansible, cloud-init, Terraform, or similar — vendor the install script into your automation repository instead of fetching it from `raw.githubusercontent.com` at provision time.

1. Download `provision.sh` (for API-key activation) or `install.sh` (for dashboard activation tokens) from a specific tagged release of the agent repository:

    ```bash
    curl -fsSL -o provision.sh \
      https://raw.githubusercontent.com/mirurobotics/agent/v0.7.3/scripts/install/provision.sh
    ```

2. Commit the downloaded script to your automation repository alongside your playbooks or modules.
3. Audit the script once. From then on, treat it as regular code in your repo — review changes through your normal pull-request process when you upgrade.
4. Call the local copy from your playbooks instead of curl-piping the upstream URL.

Benefits:

- **Pinned to a known script version.** Survives upstream script changes — your playbooks behave identically run after run.
- **No live network dependency on `raw.githubusercontent.com`.** The only network calls at provision time are to `apt.mirurobotics.com` (apt repo) and the Miru backend (activation).
- **Audit-once-then-trust.** The script becomes regular code in your repo, reviewed through your normal pull-request process.

<Info>
  Replace `v0.7.3` with the agent release you want to pin to. Browse [GitHub Releases](https://github.com/mirurobotics/agent/releases) for the full list of tagged versions.
</Info>

### Pre-bake the apt repository into a base image

For production fleets at scale — Packer pipelines, custom Yocto builds, golden Raspberry Pi or Jetson images — pre-bake the Miru apt repository into the base image and install the agent at first boot. This decomposes the wrapper script into two operations: a one-time setup at image-bake time and a per-device install at first boot.

**One-time, per image** (during image bake):

```bash
sudo apt-get install -y apt-transport-https gnupg curl ca-certificates
curl -fsSL https://apt.mirurobotics.com/miru-apt.gpg.pub \
  | sudo gpg --dearmor -o /usr/share/keyrings/miru-agent.gpg
echo "deb [signed-by=/usr/share/keyrings/miru-agent.gpg] https://apt.mirurobotics.com/agent stable main" \
  | sudo tee /etc/apt/sources.list.d/miru-agent.list
sudo apt-get update
```

**Per device** (at first boot, or via your API-key provisioning step):

```bash
sudo apt-get install -y miru-agent=<pinned-version>
sudo miru-agent --install --activation-token=$MIRU_ACTIVATION_TOKEN
```

<Info>
  `<pinned-version>` is illustrative — production fleets typically pin to a specific version for reproducibility. To get the latest version available in the repo, omit `=<pinned-version>`. Use `apt-cache madison miru-agent` on a configured machine to list available versions.

  The bare `miru-agent --install` activation defaults to the production Miru backend and MQTT hosts. For non-production environments (staging, development), pass the additional backend and MQTT host flags that `provision.sh` normally injects — see the source of [`provision.sh`](https://github.com/mirurobotics/agent/blob/main/scripts/install/provision.sh) for the exact flag names.
</Info>

Benefits:

- **Pinned versions reproducible across the fleet.** Every device that boots from the image installs exactly the same `miru-agent` version.
- **Faster first-boot.** Apt repo setup and key import happen once at image-bake time, not on every device.
- **No live curl into shell at provision time.** Every install is GPG-verified by apt against the keyring baked into the image.
- **One fewer network dependency at provisioning time.** First-boot only needs to reach `apt.mirurobotics.com` and the Miru backend.

## Poor Connectivity

Some environments may have a poor internet connection. Whether located in a remote area or using a cellular connection, downloading the Miru Agent over a network may be impractical in some environments.

In such scenarios, we recommend downloading the Miru Agent to a computer with a stable connection and then transferring it to the target machine.

While this method circumvents the need to _download_ the Miru Agent over a network, _activating_ the agent still requires some level of connectivity to register the device with the Miru servers.

<Warning>
  Some connectivity (however weak) is required to activate the agent. If a device is not connected, the agent cannot be activated.
</Warning>

### Download the Agent

On a machine with a stable internet connection that already has the Miru apt repository configured, download the agent `.deb` without installing it.

```bash
sudo apt-get update
apt-get download miru-agent
```

This produces a file named `miru-agent_<version>_<arch>.deb` in the current directory, matching the host architecture.

<Info>
  To download a specific version, pass it explicitly: `apt-get download miru-agent=<version>`. Use `apt-cache madison miru-agent` to list available versions.
</Info>

To download for a different architecture (for example a Jetson `arm64` device from an `amd64` workstation), see [GitHub Releases](https://github.com/mirurobotics/agent/releases) where every architecture's `.deb` is published as a release asset. You can find the latest release [here](https://github.com/mirurobotics/agent/releases/latest).

<Info>
  Releases with the `Pre-release` tag are not yet considered stable and may contain bugs or other issues. Use at your own risk.
</Info>

### Transfer the Agent

Once you've downloaded the appropriate Debian package, transfer it to your target machine. This can be done via a variety of methods, such as a USB drive.

### Install the Agent

After transferring the package to your target machine, install it by adding the `--from-pkg` argument to the installation command.

**API keys**

If installing via an [API key](#api-keys), add the `--from-pkg` argument to the provision script.

```bash
curl -fsSL https://raw.githubusercontent.com/mirurobotics/agent/main/scripts/install/provision.sh | sh -s -- \
--device-name=$HOSTNAME \
--allow-reactivation=true \
--from-pkg=/path/to/miru-agent_<version>_<architecture>.deb
```

<Info>
  You'll need to replace `/path/to/miru-agent_<version>_<architecture>.deb` with the actual path to the package.
</Info>

**Dashboard**

If installing via the dashboard, follow these steps.

1. Copy the [installation command](#install-the-agent) from the dashboard
2. Append `-s -- --from-pkg=/path/to/agent.deb` and replace the `<path/to/agent.deb>` with the actual path to the agent package

This will give you an installation command similar to the following.

```bash
curl -fsSL https://raw.githubusercontent.com/mirurobotics/agent/main/scripts/install/install.sh | \
env MIRU_ACTIVATION_TOKEN=<activation-token> \
sh -s -- \
--from-pkg=/path/to/agent.deb
```

<Warning>
  Do not copy the example command above. It will not work because it lacks the required activation token. Instead, follow the steps listed above.
</Warning>
