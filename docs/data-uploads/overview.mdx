Miru uploads data *off* your devices and into *your* cloud storage.

You define a small set of [**upload rules**](/data-uploads/upload-rules/overview) — what to
collect, when it is considered finished, and where it goes. The
[Miru Agent](/developers/agent/overview) detects matching data, confirms each file is
complete, and uploads it directly to your bucket.

## How it works

<img
  src="https://assets.mirurobotics.com/docs/v04/images/data-uploads/upload-flow.svg"
  alt="Upload flow: a device requests an upload from the Miru control plane, receives short-lived downscoped credentials for a single object key carrying its own device ID, and uploads the file directly to your cloud storage with the native cloud SDK. Your data never passes through Miru."
/>
{/* TODO (design): refresh upload-flow.svg — it still depicts the old signed-URL flow; the current flow vends downscoped credentials used with the native cloud SDK. */}

1. The device detects a completed file and makes an **upload request** to the Miru
   control plane.
2. The control plane verifies the device's identity, records the upload in a ledger,
   and returns **short-lived, downscoped cloud credentials** valid for exactly one
   object key — a key carrying that device's own verified ID.
3. The device uploads the file **directly to your bucket** using the native cloud SDK
   (a multipart upload for S3, a resumable upload for GCS), then confirms the durable
   write.

Your data moves straight from the device to your storage — only the upload request and
the credential exchange touch Miru. See [Uploads](/data-uploads/uploads) for the full
flow and its guarantees.
