Once an [upload rule](/data-uploads/upload-rules) is active on a device, the
[Miru Agent](/developers/agent/overview) handles the rest: detecting when a matching
file is finished, uploading it to your [bucket](/data-uploads/buckets/overview), and
recording it in the ledger with the properties below.

  <ImmutableBadge />

  - `uploaded` — the device confirmed a durable write to the bucket.
</ParamField>

<ParamField path="incomplete" type="boolean">

  <ImmutableBadge />

<ParamField path="deployment_id" type="string">
  <ImmutableBadge />

  Example: `dpl_123`
</ParamField>

  Returned on Device API responses only.

Uploads also carry the standard `created_at`, `updated_at`, and `workspace_id`
bookkeeping fields.

## How an upload happens

Devices never hold long-lived cloud credentials, and Miru never proxies your data.
Each upload is brokered with short-lived, downscoped credentials:

1. The Agent detects a completed file matching a rule and **requests an upload** from
   the Miru control plane, sending the rule ID, file details, digest, and size.
2. The control plane verifies the device's identity from its session token, records a
   `pending` ledger entry, deduplicates by digest, and vends **short-lived downscoped
   cloud credentials** scoped to exactly one object key — a key carrying that device's
   own verified ID.
3. The device uploads the file **directly to your bucket** with the native cloud SDK —
   a multipart upload for S3, a resumable upload for GCS. If the credentials expire
   mid-upload, the device requests fresh credentials for the same key and resumes.
4. The device **confirms** the durable write; the ledger entry flips to `uploaded` and
   `uploaded_at` is set.

Your data moves straight from the device to your storage — only the upload request and
the credential exchange touch Miru.

## Completion detection

The Agent uploads a file once it is confident the file has finished being written to.
Completion is determined with two complementary checks:

- **Finalization marker** — for formats that end with one (MCAP, parquet), Miru
  verifies the trailing bytes that only exist once the writer has closed the file
  cleanly. For these formats, completion is guaranteed.
- **Quiescence** — a file whose size and modification time have held steady for the
  rule's [`stability_window_secs`](/data-uploads/upload-rules#sources) is treated as
  ready. This works for any file type, but it is a heuristic: "looks done," not
  "proven done."

If you would rather not rely on detection at all, you can **defer completion to the
file producer**: have it write to a temporary name and rename on completion (or drop a
marker file), and the Agent will only ever see finished files — zero false positives.

A file that goes quiet but was never closed cleanly is still uploaded, but flagged
[`incomplete`](#properties).

## Guarantees

The following properties are guaranteed by the upload flow.

**Correctness**

- **Exactly one copy.** The ledger deduplicates by [`digest`](#properties), so each
  file is uploaded exactly once.
- **Unique keys.** Every object key contains the [upload's ID](#properties), so an
  upload can never overwrite another object.
- **Atomic visibility.** An object appears in your bucket only when fully written; a
  partial or in-flight upload is never visible to anything reading the bucket.

**Safety**

- **Non-destructive.** The credentials a device receives only allow *creating* one
  specific object — they cannot read, list, modify, or delete anything else in your
  bucket.
- **Durable before acknowledged.** A file is recorded as [`uploaded`](#properties)
  only after the device confirms a durable write to the bucket.

**Delivery**

- **Resumable.** Uploads are chunked (S3 multipart / GCS resumable) and resume where
  they left off across poor network conditions; expired credentials are re-vended
  mid-upload without restarting.

## Object metadata

Each uploaded object is stamped with the upload's provenance as cloud object metadata
(S3 `x-amz-meta-*` / GCS `x-goog-meta-*`), so every file is self-describing in your
bucket and can be correlated back to exactly what produced it without parsing the
path.

The metadata is a generic map of string keys to string values. The keys currently
stamped include `device_id`, `release_id`, `release_version`, `deployment_id`,
`digest`, and `file_modified_at`, but the set may evolve. The map stays within the
cloud provider's object-metadata size limits (roughly 2 KB on S3, 8 KB on GCS).

Uploads are records the [Miru Agent](/developers/agent/overview) produces — you don't
create or edit them by hand. The **Uploads** view lists every upload in the workspace,
