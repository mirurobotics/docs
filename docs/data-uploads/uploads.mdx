Once an [upload rule](/data-uploads/upload-rules/overview) is active on a device, the
[Miru Agent](/developers/agent/overview) handles the rest: detecting when a matching
file is finished, uploading it to your [bucket](/data-uploads/buckets/overview), and
recording it with the provenance below.

<ParamField path="size" type="integer">
  <ImmutableBadge />

  The size of the uploaded file, in bytes.
</ParamField>

<ParamField path="status" type="enum">
  <MutableBadge />

  The lifecycle status of the upload in the ledger.

  - `pending` — a presigned URL has been minted but the device has not started uploading.
  - `uploading` — the device is uploading the file to the bucket.
  - `uploaded` — the device confirmed a durable write to the bucket.
</ParamField>

<ParamField path="incomplete" type="boolean">
  <ImmutableBadge />

  Whether the file was incomplete when collected (uploaded without a confirmed
  finalization). Independent of `status`.
</ParamField>

<ParamField path="release_id" type="string">
  <ImmutableBadge />

  ID of the [release](/learn/releases/overview) the device was on when the upload was
  produced.
</ParamField>

<ParamField path="uploading_at" type="datetime">
  <MutableBadge />

  Timestamp of when the device began uploading the file. Null until the upload starts.
</ParamField>

<ParamField path="uploaded_at" type="datetime">
  <MutableBadge />

  Timestamp of when the upload was durably confirmed. Null until the upload completes.
</ParamField>

## Completion detection

The Agent uploads a file once it is confident the file has finished being written to.
Completion is determined with two complementary checks:

- **Finalization marker** — for formats that end with one (MCAP, parquet), Miru
  verifies the trailing bytes that only exist once the writer has closed the file
  cleanly. For these formats, completion is guaranteed.
- **Quiescence** — a file whose size and modification time have held steady for the
  rule's [`stability_window`](/data-uploads/upload-rules/sources) is treated as ready.
  This works for any file type, but it is a heuristic: "looks done," not "proven done."

If you would rather not rely on detection at all, you can **defer completion to the
file producer**: have it write to a temporary name and rename on completion (or drop a
marker file), and the Agent will only ever see finished files — zero false positives.

A file that goes quiet but was never closed cleanly is still uploaded, but flagged
[`incomplete`](#properties).

## Guarantees

The following properties are guaranteed by the Miru Agent.

**Correctness**

- **Exactly one copy.** Each file is uploaded once, keyed on its
  [`digest`](#properties).
- **Integrity.** Every object is verified to match its source byte-for-byte.
- **Atomic visibility.** An object appears in your bucket only when fully written; a
  partial or in-flight upload is never visible to anything reading the bucket.

**Safety**

- **Non-destructive.** Miru only ever *creates* new objects. The signing identity
  cannot modify or delete existing data.
- **Durable before acknowledged.** A file is recorded as
  [`uploaded`](#properties) only after the bucket confirms a durable write.

**Delivery**

- **Resumable.** Uploads are chunked and resume where they left off across poor
  network conditions.

## Provenance

The properties above are also stamped onto each object as storage metadata, so every
file is self-describing in your bucket and can be correlated back to exactly what
produced it — the device, rule, upload type, and release — without parsing the path.

## Viewing uploads  {/* TODO: confirm required role/scope */}

Uploads are records the [Miru Agent](/developers/agent/overview) produces — you don't
create or edit them by hand. From the **Uploads** view you can browse the ledger and
open any upload to inspect its [properties](#properties): the device, rule, upload type,
and release that produced it, its `object_key`, `digest`, `size`, lifecycle
[`status`](#properties), and whether it was flagged [`incomplete`](#completion-detection).
