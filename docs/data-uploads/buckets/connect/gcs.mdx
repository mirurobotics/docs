A Google Cloud Storage (GCS) bucket is connected via **Workload Identity Federation
(WIF)** — the native mechanism Google Cloud provides for letting an external identity
act in your project *without* a service-account key. 


## How it works

1. Miru presents its AWS identity to Google
2. Google checks it against a Workload Identity Pool you configure
3. Miru impersonates an service account in your project
4. Miru mints **short-lived access tokens** to access your bucket

<img 
  src="/images/gcs-wif-token-exchange.svg" 
  alt="GCS WIF token exchange" 
/>

Connecting your bucket to Miru is a one-time setup that uses only configuration values. No service-account keys, passwords, or other secrets change hands in either direction. 

### Miru provides you

<ParamField path="miru_aws_account_id">
  The AWS account ID Miru uses to federate into your GCP project. You grant this account access to your bucket. This is a fixed value that never changes.

  `008971655189`
</ParamField>

<ParamField path="miru_aws_role_arn">
  The identity-only AWS role Miru uses to federate into your GCP project. You grant this role access to your bucket. This is a fixed value that never changes.

  `arn:aws:iam::008971655189:role/MiruGCSIntegrationRole`
</ParamField>

<ParamField path="workspace_id">
  Your Miru workspace ID. Miru sets this as the STS **session name** when it assumes its integration role, and your Workload Identity provider must pin it (below) so only Miru acting for your workspace can federate. Shown in the Miru dashboard.

  Example: `wsp_123`
</ParamField>

### You provide Miru

<ParamField path="bucket_name">
  <ImmutableBadge />

  The name of your GCS bucket to connect to Miru.

  Example: `my-upload-bucket`
</ParamField>

<ParamField path="wip_provider">
  <MutableBadge />

  The full Workload Identity Pool provider resource name used to federate Miru's
  identity into your GCP project.

  Example: `projects/123456789/locations/global/workloadIdentityPools/miru-pool/providers/miru-provider`
</ParamField>

<ParamField path="service_account_email">
  <MutableBadge />

  The email of the GCP service account Miru impersonates to access your bucket.

  Example: `miru@my-gcp-project.iam.gserviceaccount.com`
</ParamField>

## Connect a bucket

Follow the steps below to connect a bucket to Miru using the Google Cloud CLI.
To connect a bucket to Miru, you must hold the <PublisherBadge size="sm"/> role.

<Steps>
  <Step title="Get your workspace ID from Miru">
    In the Miru dashboard, copy your **workspace ID** and set it as a variable. Your
    Workload Identity provider will pin to this value — Miru presents it as the STS
    session name when it federates, so only Miru acting for your workspace can assume
    the identity.

    ```bash
    WORKSPACE_ID="<workspace-id>"
    ```
  </Step>

  <Step title="Define the bucket">
    Set your Google Cloud project ID and bucket variables.

    ```bash
    PROJECT_ID="<project-id>"
    BUCKET_NAME="<bucket-name>"
    BUCKET_LOCATION="<region>"
    ```

    Create the bucket if it doesn't yet exist.

    ```bash
    gcloud storage buckets create "gs://${BUCKET_NAME}" \
      --project="${PROJECT_ID}" \
      --location="${BUCKET_LOCATION}"
    ```
  </Step>

  <Step title="Create a Workload Identity Pool">
    The pool is the container that holds the AWS provider Miru federates through.

    ```bash
    WIP_ID="miru-pool"
    gcloud iam workload-identity-pools create "${WIP_ID}" \
      --project="${PROJECT_ID}" \
      --location=global \
      --display-name="Miru pool"
    ```
  </Step>

  <Step title="Add an AWS provider that pins Miru's role">
    
    The attribute mapping makes the federated subject the caller's full assumed-role
    ARN — including the session name — and the attribute condition accepts **only
    Miru's exact role AND your workspace's session name**. Pinning the session name is
    what blocks another Miru workspace from federating through a copy of this
    configuration (the confused-deputy guard): a caller presenting Miru's role but a
    different session name is rejected.

    ```bash
    WIP_PROVIDER_ID="miru-provider"
    MIRU_AWS_ACCOUNT_ID="008971655189"
    MIRU_ASSUMED_ROLE="arn:aws:sts::${MIRU_AWS_ACCOUNT_ID}:assumed-role/MiruGCSIntegrationRole"
    gcloud iam workload-identity-pools providers create-aws "${WIP_PROVIDER_ID}" \
      --project="${PROJECT_ID}" \
      --location=global \
      --workload-identity-pool="${WIP_ID}" \
      --account-id="${MIRU_AWS_ACCOUNT_ID}" \
      --attribute-mapping="google.subject=assertion.arn" \
      --attribute-condition="assertion.arn == '${MIRU_ASSUMED_ROLE}/${WORKSPACE_ID}'"
    ```

    Save the provider's full resource name for bucket registration:

    ```bash
    WIP_PROVIDER="$(gcloud iam workload-identity-pools providers describe \
      "${WIP_PROVIDER_ID}" \
      --project="${PROJECT_ID}" \
      --location=global \
      --workload-identity-pool="${WIP_ID}" \
      --format="value(name)")"
    ```
  </Step>

  <Step title="Create the service account">
    Miru impersonates this service account to mint the downscoped tokens devices
    upload with. Its email
    (`${SERVICE_ACCOUNT_EMAIL}`) becomes your `service_account_email`.

    ```bash
    SERVICE_ACCOUNT_ID="miru-uploader"
    gcloud iam service-accounts create "${SERVICE_ACCOUNT_ID}" \
      --project="${PROJECT_ID}" \
      --display-name="Miru uploader"
    SERVICE_ACCOUNT_EMAIL="${SERVICE_ACCOUNT_ID}@${PROJECT_ID}.iam.gserviceaccount.com"
    ```
  </Step>

  <Step title="Let the service account create objects in the bucket">
    Grant the uploader service account permission to create objects on the bucket.

    ```bash
    gcloud storage buckets add-iam-policy-binding "gs://${BUCKET_NAME}" \
      --member="serviceAccount:${SERVICE_ACCOUNT_EMAIL}" \
      --role="roles/storage.objectCreator"
    ```

    <Note>
      [Verification](/data-uploads/buckets/manage#verify-a-bucket) writes a
      zero-byte probe object under `.miru/probe/`.
    </Note>
  </Step>

  <Step title="Let Miru's federated principal impersonate the service account">
    Bind the **Workload Identity User** role
    (`roles/iam.workloadIdentityUser`) on the uploader service account as a
    **per-subject** binding on the exact session
    (`principal://.../subject/<assumed-role ARN>/<workspace id>`). This lets only
    Miru's pinned session mint short-lived access tokens as the uploader account —
    and nothing else.

    ```bash
    PROJECT_NUMBER="$(gcloud projects describe "${PROJECT_ID}" --format='value(projectNumber)')"
    gcloud iam service-accounts add-iam-policy-binding \
      "${SERVICE_ACCOUNT_EMAIL}" \
      --project="${PROJECT_ID}" \
      --role="roles/iam.workloadIdentityUser" \
      --member="principal://iam.googleapis.com/projects/${PROJECT_NUMBER}/locations/global/workloadIdentityPools/${WIP_ID}/subject/${MIRU_ASSUMED_ROLE}/${WORKSPACE_ID}"
    ```
  </Step>

  <Step title="Register the bucket in Miru">
    Print the values needed to register the bucket:

    ```bash
    printf 'name: %s\nwip_provider: %s\nservice_account_email: %s\n' \
      "${BUCKET_NAME}" \
      "${WIP_PROVIDER}" \
      "${SERVICE_ACCOUNT_EMAIL}"
    ```

    From the **Buckets** page in the Miru dashboard, select **New bucket**, choose
    **GCS**, and supply:

    - `bucket_name` — the name of the GCS bucket
    - `wip_provider` — the Workload Identity Pool provider resource name
    - `service_account_email` — the uploader service account's email

    Select **Create** to register the bucket.

</Steps>
