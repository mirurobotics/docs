A GCS bucket is authenticated via **Workload Identity Federation (WIF)** — the
native mechanism Google Cloud provides for letting an external identity act in your
project *without* a service-account key. Miru presents its AWS identity to Google,
Google checks it against a Workload Identity Pool you configure, and Miru impersonates
an uploader service account to mint **short-lived access tokens downscoped to a single
object**; the device uploads directly with that token. No service-account key is ever
downloaded. See [Connecting your bucket](#connecting-your-bucket) for the
click-by-click setup.

## Properties

The top-level bucket [`name`](/data-uploads/buckets/overview#properties) is the name
of the GCS bucket itself. The GCS-specific `config` adds:

<ParamField path="project_id" type="string" required>
  <ImmutableBadge />

  The GCP project ID that owns the bucket and the Workload Identity Federation
  configuration.
</ParamField>

<ParamField path="wip_provider" type="string" required>
  <ImmutableBadge />

  The full Workload Identity Pool provider resource name used to federate Miru's
  identity into your GCP project.

  Example: `projects/123456789/locations/global/workloadIdentityPools/miru-pool/providers/miru-provider`
</ParamField>

<ParamField path="service_account_email" type="string" required>
  <ImmutableBadge />

  The email of the GCP service account Miru impersonates to mint short-lived,
  downscoped upload tokens.

  Example: `miru-uploader@my-gcp-project.iam.gserviceaccount.com`
</ParamField>

## Connecting your bucket

No secrets change hands in either direction. The integration is established by
exchanging **identifiers and configuration only**:

| Miru provides to you | You provide to Miru |
| --- | --- |
| Miru's AWS account ID | Your GCP project ID |
| Miru's IAM role ARN (`<miru-aws-role-arn>`) | The bucket name |
| | The Workload Identity Pool provider resource name |
| | The uploader service account email |

There are **no service-account keys, passwords, or long-lived tokens** in this list —
by design. Nothing you hand Miru can be replayed or exfiltrated to reach your data.

The steps below are one-time setup in **your own GCP project**. Each gives the GCP
Console path and the equivalent `gcloud` command. Throughout, replace `<…>`
placeholders with your own values; `<miru-aws-role-arn>` is the role ARN Miru gives
you at registration.

<Steps>
  <Step title="Create or choose the GCS bucket">
    Pick (or create) the bucket uploads should land in. Note its **name** and the
    **project** that owns it.

    Console: **Cloud Storage → Buckets → Create**. Pick a name, region, and storage
    class, then create.

    ```bash
    gcloud storage buckets create gs://<bucket-name> \
      --project=<project-id> \
      --location=<region>
    ```

    <Frame>
      ![Cloud Storage Buckets list with the Create button highlighted](https://assets.mirurobotics.com/docs/v04/images/data-uploads/buckets/gcs/create-bucket.png)
    </Frame>
    {/* TODO (screenshot): GCP Console, Cloud Storage → Buckets list, highlight the "Create" button (or the target bucket row if reusing an existing bucket). */}
  </Step>

  <Step title="Create a Workload Identity Pool">
    The pool is the container that holds the AWS provider Miru federates through.

    Console: **IAM & Admin → Workload Identity Federation → Create Pool**. Give it a
    name (for example `miru-pool`) and create it.

    ```bash
    gcloud iam workload-identity-pools create miru-pool \
      --project=<project-id> \
      --location=global \
      --display-name="Miru pool"
    ```

    <Frame>
      ![Workload Identity Federation page with the Create Pool form open](https://assets.mirurobotics.com/docs/v04/images/data-uploads/buckets/gcs/create-pool.png)
    </Frame>
    {/* TODO (screenshot): GCP Console, IAM & Admin → Workload Identity Federation → Create Pool form, highlight the pool name field and the "Continue" button. */}
  </Step>

  <Step title="Add an AWS provider that pins Miru's role">
    Add an **AWS** provider to the pool and set its **attribute condition** to accept
    **only Miru's exact role**. This is the linchpin: a copy of the configuration is
    useless to anyone who is not presenting Miru's specific AWS identity.

    Console: in the pool, **Add Provider → Provider type: AWS**, enter Miru's AWS
    account ID, then under **Attribute conditions** add a condition pinning the
    assumed-role ARN Miru gives you.

    ```text
    google.subject == '<miru-aws-role-arn>'
    ```

    {/* TODO (product): confirm the exact attribute condition expression and which CEL attribute is matched (e.g. assertion.arn / attribute.aws_role / google.subject) and the precise assumed-role ARN string format. */}

    ```bash
    gcloud iam workload-identity-pools providers create-aws miru-provider \
      --project=<project-id> \
      --location=global \
      --workload-identity-pool=miru-pool \
      --account-id=<miru-aws-account-id> \
      --attribute-condition="google.subject == '<miru-aws-role-arn>'"
    ```

    The provider's full resource name is your `wip_provider` value:
    `projects/<project-number>/locations/global/workloadIdentityPools/miru-pool/providers/miru-provider`.

    <Frame>
      ![Add provider form with provider type AWS and the attribute condition pinning Miru's role](https://assets.mirurobotics.com/docs/v04/images/data-uploads/buckets/gcs/add-aws-provider.png)
    </Frame>
    {/* TODO (screenshot): GCP Console, pool's Add Provider form, provider type AWS, highlight the AWS account ID field and the Attribute conditions box with the pinned role ARN. */}
  </Step>

  <Step title="Create the uploader service account">
    Miru impersonates this service account to mint the downscoped tokens devices
    upload with.

    Console: **IAM & Admin → Service Accounts → Create Service Account**. Name it (for
    example `miru-uploader`) and create. Its email becomes your
    `service_account_email`.

    ```bash
    gcloud iam service-accounts create miru-uploader \
      --project=<project-id> \
      --display-name="Miru uploader"
    ```

    <Frame>
      ![Create Service Account form with the name miru-uploader](https://assets.mirurobotics.com/docs/v04/images/data-uploads/buckets/gcs/create-service-account.png)
    </Frame>
    {/* TODO (screenshot): GCP Console, IAM & Admin → Service Accounts → Create Service Account, highlight the service account name field and the generated email. */}
  </Step>

  <Step title="Let the service account create objects in the bucket">
    Grant the uploader service account permission to create objects on the bucket —
    optionally scoped with an IAM condition so it can only ever write under a key
    prefix.

    Console: on the bucket, **Permissions → Grant Access**. Set the principal to the
    service account email and the role to **Storage Object Creator**
    (`roles/storage.objectCreator`).

    ```bash
    gcloud storage buckets add-iam-policy-binding gs://<bucket-name> \
      --member="serviceAccount:miru-uploader@<project-id>.iam.gserviceaccount.com" \
      --role="roles/storage.objectCreator"
    ```

    <Note>
      [Verification](/data-uploads/buckets/manage#verify-a-bucket) writes a
      zero-byte probe object under `.miru/probe/` and then attempts to delete it.
      Cleanup is best-effort — verification succeeds without it — but if you want the
      probe objects removed, also grant a delete-capable role scoped to the
      `.miru/probe/` prefix with an IAM condition.
    </Note>

    <Frame>
      ![Bucket Permissions tab with Grant Access granting Storage Object Creator to the uploader service account](https://assets.mirurobotics.com/docs/v04/images/data-uploads/buckets/gcs/grant-object-creator.png)
    </Frame>
    {/* TODO (screenshot): GCP Console, bucket → Permissions → Grant Access dialog, highlight the principal (service account email) and the Storage Object Creator role selection. */}
  </Step>

  <Step title="Let Miru's federated principal impersonate the service account">
    Bind the **Service Account Token Creator** role
    (`roles/iam.serviceAccountTokenCreator`) on the uploader service account to the
    principal that represents Miru's pinned AWS role via the pool. This lets the
    federated identity mint short-lived access tokens as the uploader account — and
    nothing else.

    Console: **Service Accounts → (the uploader SA) → Permissions → Grant Access**.
    Set the principal to the pool principal set and the role to **Service Account
    Token Creator**.

    ```bash
    gcloud iam service-accounts add-iam-policy-binding \
      miru-uploader@<project-id>.iam.gserviceaccount.com \
      --project=<project-id> \
      --role="roles/iam.serviceAccountTokenCreator" \
      --member="principalSet://iam.googleapis.com/projects/<project-number>/locations/global/workloadIdentityPools/miru-pool/*"
    ```

    {/* TODO (product): confirm the exact principalSet selector — whole pool (`/*`) vs. a single subject (`/subject/<miru-aws-role-arn>`) — and that token creation is the only required IAM permission on the SA. */}

    <Frame>
      ![Service account Permissions tab granting Service Account Token Creator to the pool principal set](https://assets.mirurobotics.com/docs/v04/images/data-uploads/buckets/gcs/grant-token-creator.png)
    </Frame>
    {/* TODO (screenshot): GCP Console, uploader service account → Permissions → Grant Access dialog, highlight the principalSet member and the Service Account Token Creator role. */}
  </Step>

  <Step title="Register the bucket in Miru">
    From the **Buckets** page in Miru, select **New bucket**, choose **GCS**, and
    supply:

    - `name` — the bucket from step 1
    - `project_id` — your GCP project ID
    - `wip_provider` — the provider resource name from step 3
    - `service_account_email` — the uploader SA email from step 4

    Select **Create** to register the bucket.

    {/* TODO (product): confirm where registration surfaces — dashboard or CLI — and document the concrete flow here. */}

    <Frame>
      ![Miru New bucket form for the GCS provider with the four fields](https://assets.mirurobotics.com/docs/v04/images/data-uploads/buckets/gcs/register-bucket.png)
    </Frame>
    {/* TODO (screenshot): Miru dashboard, Buckets → New bucket form with provider set to GCS, highlight the name, project_id, wip_provider, and service_account_email fields. */}
  </Step>

  <Step title="Verify the bucket">
    Run **Verify** on the bucket to exercise the full federation and impersonation
    chain end to end. The bucket's
    [status](/data-uploads/buckets/overview#properties) flips to `verified` on
    success, or `invalid` with the failure reason in `verification_error`.

    <Frame>
      ![Miru bucket page showing the Verify action and a verified status](https://assets.mirurobotics.com/docs/v04/images/data-uploads/buckets/gcs/verify-bucket.png)
    </Frame>
    {/* TODO (screenshot): Miru dashboard, bucket detail/menu with the Verify action and the resulting verified status badge. */}
  </Step>
</Steps>

### How device credentials are minted at runtime

When a device requests an upload, the Miru control plane performs this exchange with
Google Cloud. The device only ever sees the final token.

The chain is keyless — no service-account key is ever downloaded."
/>

1. The Miru control plane assumes its **AWS role**.
2. It presents that identity to **Google STS**, which checks it against your pool's
   attribute condition and returns a **short-lived federated Google token**.
3. Using that token, the control plane impersonates your uploader service account to
   mint a short-lived access token, and **downscopes** it with a Credential Access
   Boundary to creating **one specific object** — a key carrying the requesting
   device's own verified ID.
4. The device uses the downscoped token to drive a native **resumable upload**
   directly to your bucket; if it expires mid-upload, the device requests a fresh
   token for the same key and resumes.

The chain is keyless end to end — no service-account key is ever downloaded.
