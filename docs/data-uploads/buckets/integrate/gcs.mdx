A GCS bucket is authenticated via **Workload Identity Federation (WIF)** — the
native mechanism Google Cloud provides for letting an external identity act in your
project *without* a service-account key. Miru presents its AWS identity to Google,
Google checks it against a Workload Identity Pool you configure, and Miru impersonates
an uploader service account to mint **short-lived access tokens downscoped to a single
object**; the device uploads directly with that token. No service-account key is ever
downloaded. See [Connecting your bucket](#connecting-your-bucket) for the
click-by-click setup.

## Config

These are the fields of the bucket's
[`config`](/data-uploads/buckets/overview#properties) property when the bucket's
`provider` is `gcs`. The top-level bucket
[`name`](/data-uploads/buckets/overview#properties) is the name of the GCS bucket
itself; `config` carries everything provider-specific:

<ParamField path="provider" type="string" required>
  <ImmutableBadge />

  The provider discriminator, always `gcs` for this shape.
</ParamField>

<ParamField path="project_id" type="string" required>
  <ImmutableBadge />

  The GCP project ID that owns the bucket and the Workload Identity Federation
  configuration.
</ParamField>

<ParamField path="wip_provider" type="string" required>
  <ImmutableBadge />

  The full Workload Identity Pool provider resource name used to federate Miru's
  identity into your GCP project.

  Example: `projects/123456789/locations/global/workloadIdentityPools/miru-pool/providers/miru-provider`
</ParamField>

<ParamField path="service_account_email" type="string" required>
  <ImmutableBadge />

  The email of the GCP service account Miru impersonates to mint short-lived,
  downscoped upload tokens.

  Example: `miru-uploader@my-gcp-project.iam.gserviceaccount.com`
</ParamField>

## Connecting your bucket

No secrets change hands in either direction. The integration is established by
exchanging **identifiers and configuration only**:

| Miru provides to you | You provide to Miru |
| --- | --- |
| Miru's AWS account ID | Your GCP project ID |
| Miru's IAM role ARN (`<miru-aws-role-arn>`) | The bucket name |
| | The Workload Identity Pool provider resource name |
| | The uploader service account email |

There are **no service-account keys, passwords, or long-lived tokens** in this list —
by design. Nothing you hand Miru can be replayed or exfiltrated to reach your data.

The setup below is one-time, in **your own GCP project**, and can be done three ways:
the **Console** tab walks through the GCP Console click-by-click with the equivalent
`gcloud` commands; the **CLI** tab runs the same setup as a plain `gcloud` command
sequence; the **Terraform** tab provisions the same resources as a single
infrastructure-as-code config. Throughout, replace `<…>` placeholders with your own
values; `<miru-aws-role-arn>` is the assumed-role ARN Miru gives you at registration
(`arn:aws:sts::<miru-aws-account-id>:assumed-role/…`).

      <Step title="Create or choose the GCS bucket">
        Pick (or create) the bucket uploads should land in. Note its **name** and the
        **project** that owns it.

        Console: **Cloud Storage → Buckets → Create**. Pick a name, region, and

        ```bash
          --project=<project-id> \
          --location=<region>
        ```

      <Step title="Create a Workload Identity Pool">
        The pool is the container that holds the AWS provider Miru federates through.

        Console: **IAM & Admin → Workload Identity Federation → Create Pool**. Give
        it a name (for example `miru-pool`) and create it.

        ```bash
        gcloud iam workload-identity-pools create miru-pool \
          --project=<project-id> \
          --location=global \
          --display-name="Miru pool"
        ```

        <Frame>
          ![Workload Identity Federation page with the Create Pool form open](https://assets.mirurobotics.com/docs/v04/images/data-uploads/buckets/gcs/create-pool.png)
        </Frame>
        {/* TODO (screenshot): GCP Console, IAM & Admin → Workload Identity Federation → Create Pool form, highlight the pool name field and the "Continue" button. */}
      </Step>

      <Step title="Add an AWS provider that pins Miru's role">
        Add an **AWS** provider to the pool. Set its **attribute mapping** so the
        federated subject is the caller's ARN, plus an `attribute.aws_role` mapping
        that strips the per-session suffix from the assumed-role ARN so the
        impersonation binding can target the role itself:

        ```text
        google.subject = assertion.arn
        attribute.aws_role = assertion.arn.contains('assumed-role') ? assertion.arn.extract('{account_arn}assumed-role/') + 'assumed-role/' + assertion.arn.extract('assumed-role/{role_name}/') : assertion.arn
        ```

        Then set its **attribute condition** to accept **only Miru's exact role**.
        This is the linchpin: a copy of the configuration is useless to anyone who is
        not presenting Miru's specific AWS identity. Miru presents an assumed-role
        ARN that carries a per-session suffix, so pin the role with a prefix match
        rather than an equality check:

        ```text
        assertion.arn.startsWith('<miru-aws-role-arn>/')
        ```

        Console: in the pool, **Add Provider → Provider type: AWS**, enter Miru's AWS
        account ID, then set the attribute mapping and the attribute condition above.

        ```bash
        gcloud iam workload-identity-pools providers create-aws miru-provider \
          --project=<project-id> \
          --location=global \
          --workload-identity-pool=miru-pool \
          --account-id=<miru-aws-account-id> \
          --attribute-mapping="google.subject=assertion.arn,attribute.aws_role=assertion.arn.contains('assumed-role') ? assertion.arn.extract('{account_arn}assumed-role/') + 'assumed-role/' + assertion.arn.extract('assumed-role/{role_name}/') : assertion.arn" \
          --attribute-condition="assertion.arn.startsWith('<miru-aws-role-arn>/')"
        ```

        The provider's full resource name is your `wip_provider` value:
        `projects/<project-number>/locations/global/workloadIdentityPools/miru-pool/providers/miru-provider`.

        <Frame>
          ![Add provider form with provider type AWS and the attribute condition pinning Miru's role](https://assets.mirurobotics.com/docs/v04/images/data-uploads/buckets/gcs/add-aws-provider.png)
        </Frame>
        {/* TODO (screenshot): GCP Console, pool's Add Provider form, provider type AWS, highlight the AWS account ID field and the Attribute conditions box with the pinned role ARN. */}
      </Step>

      <Step title="Create the uploader service account">
        Miru impersonates this service account to mint the downscoped tokens devices
        upload with.

        Console: **IAM & Admin → Service Accounts → Create Service Account**. Name it
        (for example `miru-uploader`) and create. Its email becomes your
        `service_account_email`.

        ```bash
        gcloud iam service-accounts create miru-uploader \
          --project=<project-id> \
          --display-name="Miru uploader"
        ```

        <Frame>
          ![Create Service Account form with the name miru-uploader](https://assets.mirurobotics.com/docs/v04/images/data-uploads/buckets/gcs/create-service-account.png)
        </Frame>
        {/* TODO (screenshot): GCP Console, IAM & Admin → Service Accounts → Create Service Account, highlight the service account name field and the generated email. */}
      </Step>

      <Step title="Let the service account create objects in the bucket">
        Grant the uploader service account permission to create objects on the bucket
        — optionally scoped with an IAM condition so it can only ever write under a
        key prefix.

        Console: on the bucket, **Permissions → Grant Access**. Set the principal to
        the service account email and the role to **Storage Object Creator**
        (`roles/storage.objectCreator`).

        ```bash
        gcloud storage buckets add-iam-policy-binding gs://<bucket-name> \
          --member="serviceAccount:miru-uploader@<project-id>.iam.gserviceaccount.com" \
          --role="roles/storage.objectCreator"
        ```

        <Note>
          [Verification](/data-uploads/buckets/manage#verify-a-bucket) writes a
          zero-byte probe object under `.miru/probe/` and then attempts to delete it.
          Cleanup is best-effort — verification succeeds without it — but if you want
          the probe objects removed, also grant a delete-capable role scoped to the
          `.miru/probe/` prefix with an IAM condition.
        </Note>

        <Frame>
          ![Bucket Permissions tab with Grant Access granting Storage Object Creator to the uploader service account](https://assets.mirurobotics.com/docs/v04/images/data-uploads/buckets/gcs/grant-object-creator.png)
        </Frame>
        {/* TODO (screenshot): GCP Console, bucket → Permissions → Grant Access dialog, highlight the principal (service account email) and the Storage Object Creator role selection. */}
      </Step>

      <Step title="Let Miru's federated principal impersonate the service account">
        Bind the **Workload Identity User** role (`roles/iam.workloadIdentityUser`)
        on the uploader service account to the principal set that represents Miru's
        pinned AWS role via the pool's `aws_role` attribute. This lets the federated
        identity mint short-lived access tokens as the uploader account — and nothing
        else.

        Console: **Service Accounts → (the uploader SA) → Permissions → Grant
        Access**. Set the principal to the principal set below and the role to
        **Workload Identity User**.

        ```bash
        gcloud iam service-accounts add-iam-policy-binding \
          miru-uploader@<project-id>.iam.gserviceaccount.com \
          --project=<project-id> \
          --role="roles/iam.workloadIdentityUser" \
          --member="principalSet://iam.googleapis.com/projects/<project-number>/locations/global/workloadIdentityPools/miru-pool/attribute.aws_role/<miru-aws-role-arn>"
        ```

        <Frame>
          ![Service account Permissions tab granting Workload Identity User to the pool principal set](https://assets.mirurobotics.com/docs/v04/images/data-uploads/buckets/gcs/grant-workload-identity-user.png)
        </Frame>

      </Step>

      <Step title="Write and apply the Terraform config">
        Save the config below in a new or existing workspace; it assumes an
        authenticated `google` provider is already configured. It creates
        everything the Console tab does: the bucket, the uploader service
        account and its object-creation grant, the Workload Identity Pool with

        The variables up top are the only knobs; the outputs at the bottom

        ```hcl
        variable "project_id" {
          description = "GCP project ID that owns the bucket and the Workload Identity Federation resources"
          type        = string
        }

        variable "bucket_name" {
          description = "Name of the GCS bucket device uploads land in"
          type        = string
        }

        variable "bucket_location" {
          description = "Location of the bucket (region or multi-region)"
          type        = string
          default     = "US"
        }

        variable "pool_id" {
          description = "ID for the Workload Identity Pool"
          type        = string
          default     = "miru-pool"
        }

        variable "provider_id" {
          description = "ID for the pool's AWS provider"
          type        = string
          default     = "miru-provider"
        }

        variable "service_account_id" {
          description = "Account ID for the uploader service account"
          type        = string
          default     = "miru-uploader"
        }

        variable "miru_aws_account_id" {
          description = "Miru's AWS account ID (provided by Miru)"
          type        = string
        }

        variable "miru_aws_role_arn" {
          description = "Assumed-role ARN of Miru's integration identity (provided by Miru)"
          type        = string
        }

        resource "google_storage_bucket" "uploads" {
          name                        = var.bucket_name
          project                     = var.project_id
          location                    = var.bucket_location
          uniform_bucket_level_access = true
        }

        resource "google_service_account" "miru_uploader" {
          project      = var.project_id
          account_id   = var.service_account_id
          display_name = "Miru uploader"
        }

        # Create-only: the uploader can write objects but never read, list,
        # or delete your data.
        resource "google_storage_bucket_iam_member" "miru_uploader_creator" {
          bucket = google_storage_bucket.uploads.name
          role   = "roles/storage.objectCreator"
          member = "serviceAccount:${google_service_account.miru_uploader.email}"
        }

        resource "google_iam_workload_identity_pool" "miru" {
          project                   = var.project_id
          workload_identity_pool_id = var.pool_id
        }

        # attribute.aws_role strips the per-session suffix from the
        # assumed-role ARN; the attribute condition accepts only Miru's
        # exact role.
        resource "google_iam_workload_identity_pool_provider" "miru_aws" {
          project                            = var.project_id
          workload_identity_pool_id          = google_iam_workload_identity_pool.miru.workload_identity_pool_id
          workload_identity_pool_provider_id = var.provider_id

          attribute_mapping = {
            "google.subject"     = "assertion.arn"
            "attribute.aws_role" = "assertion.arn.contains('assumed-role') ? assertion.arn.extract('{account_arn}assumed-role/') + 'assumed-role/' + assertion.arn.extract('assumed-role/{role_name}/') : assertion.arn"
          }

          attribute_condition = "assertion.arn.startsWith('${var.miru_aws_role_arn}/')"

          aws {
            account_id = var.miru_aws_account_id
          }
        }

        resource "google_service_account_iam_member" "miru_wif_impersonation" {
          service_account_id = google_service_account.miru_uploader.name
          role               = "roles/iam.workloadIdentityUser"
          member             = "principalSet://iam.googleapis.com/${google_iam_workload_identity_pool.miru.name}/attribute.aws_role/${var.miru_aws_role_arn}"
        }

        # The four fields the Miru registration form asks for, 1:1.
        output "name" {
          description = "Bucket name — the form's `name` field"
          value       = google_storage_bucket.uploads.name
        }

        output "project_id" {
          description = "GCP project ID — the form's `project_id` field"
          value       = var.project_id
        }

        output "wip_provider" {
          description = "Workload Identity Pool provider resource name — the form's `wip_provider` field"
          value       = google_iam_workload_identity_pool_provider.miru_aws.name
        }

        output "service_account_email" {
          description = "Uploader service account email — the form's `service_account_email` field"
          value       = google_service_account.miru_uploader.email
        }
        ```

### How device credentials are minted at runtime

When a device requests an upload, the Miru control plane performs this exchange with
Google Cloud. The device only ever sees the final token.

  src="https://assets.mirurobotics.com/docs/v04/images/data-uploads/credential-mint.svg"
  alt="Credential minting: the Miru control plane assumes its AWS role, presents that identity to Google STS (which checks it against your Workload Identity pool's attribute condition) for a short-lived federated token, impersonates your uploader service account, and downscopes the resulting access token to a single object. The chain is keyless — no service-account key is ever downloaded."
/>
{/* TODO (design): refresh credential-mint.svg — it still depicts V4 signed URLs; the final step now downscopes an access token with a Credential Access Boundary instead of signing a URL. */}

1. The Miru control plane assumes its **AWS role**.
2. It presents that identity to **Google STS**, which checks it against your pool's
   attribute condition and returns a **short-lived federated Google token**.
3. Using that token, the control plane impersonates your uploader service account to
   mint a short-lived access token, and **downscopes** it with a Credential Access
   Boundary to creating **one specific object** — a key carrying the requesting
   device's own verified ID.
4. The device uses the downscoped token to drive a native **resumable upload**
   directly to your bucket; if it expires mid-upload, the device requests a fresh
   token for the same key and resumes.

The chain is keyless end to end — no service-account key is ever downloaded.
