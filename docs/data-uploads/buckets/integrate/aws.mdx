An S3 bucket is authenticated by **assuming a cross-account IAM role** (STS
AssumeRole) — there are **no access keys**. You create a role in your own AWS account
whose trust policy allows Miru's integration role to assume it, gated by an external
ID, and grant that role write access to your bucket. At upload time Miru assumes the
role and vends the device **short-lived session credentials downscoped to a single
object key**; the device uploads directly with the native S3 SDK. See
[Connecting your bucket](#connecting-your-bucket) for the click-by-click setup.

## Config

These are the fields of the bucket's
[`config`](/data-uploads/buckets/overview#properties) property when the bucket's
`provider` is `aws`. The top-level bucket
[`name`](/data-uploads/buckets/overview#properties) is the name of the S3 bucket
itself; `config` carries everything provider-specific:

<ParamField path="provider" type="string" required>
  <ImmutableBadge />

  The provider discriminator, always `aws` for this shape.
</ParamField>

<ParamField path="region" type="string" required>
  <ImmutableBadge />

  The AWS region of the bucket, used for the S3 endpoint and SigV4 signing.

  Example: `us-east-1`
</ParamField>

<ParamField path="role_arn" type="string" required>
  <ImmutableBadge />

  ARN of the IAM role in your account that Miru assumes to broker uploads. Its trust
  policy must allow Miru's integration role with the external ID, and it must grant
  write access on the bucket.

  Example: `arn:aws:iam::123456789012:role/miru-uploader`
</ParamField>

<ParamField path="external_id" type="string">
  <ImmutableBadge />

  The external ID that must be present in the role's trust policy, guarding against
  the confused-deputy problem. **Issued by Miru** when you register the bucket — you
  never supply it, you copy it into your trust policy.

  Example: `miru-a1b2c3d4`
</ParamField>

## Connecting your bucket

No secrets change hands in either direction. The integration is established by
exchanging **identifiers and configuration only** — Miru never holds an access key to
your account:

| Miru provides to you | You provide to Miru |
| --- | --- |
| Miru's integration role ARN (`<miru-integration-role-arn>`) | The bucket name |
| An external ID, issued when you register the bucket | The bucket region |
| | The IAM role ARN Miru may assume |

There are **no access keys, passwords, or long-lived tokens** in this exchange — by
design. Nothing you hand Miru can be replayed or exfiltrated to reach your data.

The steps below are one-time setup, mostly in **your own AWS account**. Each gives the
AWS Console path, the equivalent AWS CLI command, and literal policy JSON. Replace
`<…>` placeholders with your own values; `<miru-integration-role-arn>` is the value
Miru gives you, and the external ID is issued when you register the bucket in step 4.

<Steps>
  <Step title="Create or choose the S3 bucket">
    Pick (or create) the bucket uploads should land in. Note its **name** and
    **region**.

    Console: **S3 → Create bucket**. Pick a name and region, then create.

    ```bash
    aws s3api create-bucket \
      --bucket <bucket-name> \
      --region <region> \
      --create-bucket-configuration LocationConstraint=<region>
    ```

    <Frame>
      ![S3 Create bucket page with the bucket name and region fields](https://assets.mirurobotics.com/docs/v04/images/data-uploads/buckets/aws/create-bucket.png)
    </Frame>
    {/* TODO (screenshot): AWS Console, S3 → Create bucket, highlight the bucket name and AWS Region fields (or the target bucket row in the S3 buckets list if reusing one). */}
  </Step>

  <Step title="Create a cross-account IAM role">
    Create an IAM role whose **trust policy** allows **Miru's integration role** as
    the principal. You will tighten this policy with the external ID in step 5, once
    Miru has issued it.

    Console: **IAM → Roles → Create role → Custom trust policy**, and paste:

    ```json
    {
      "Version": "2012-10-17",
      "Statement": [
        {
          "Effect": "Allow",
          "Principal": { "AWS": "<miru-integration-role-arn>" },
          "Action": "sts:AssumeRole"
        }
      ]
    }
    ```

    CLI: save the JSON above as `trust-policy.json`, then:

    ```bash
    aws iam create-role \
      --role-name miru-uploader \
      --assume-role-policy-document file://trust-policy.json
    ```

    The role's ARN (`arn:aws:iam::<your-account-id>:role/miru-uploader`) is your
    `role_arn`.

    <Frame>
      ![IAM Create role page with a custom trust policy pasted in, showing the Miru integration role principal](https://assets.mirurobotics.com/docs/v04/images/data-uploads/buckets/aws/trust-policy.png)
    </Frame>
    {/* TODO (screenshot): AWS Console, IAM → Roles → Create role → Custom trust policy editor, highlight the Principal (Miru's integration role ARN). */}
  </Step>

  <Step title="Grant the role write access to the bucket">
    Attach a permissions policy granting **`s3:PutObject`** and
    **`s3:AbortMultipartUpload`** on the bucket — devices upload via native S3
    multipart uploads, and the abort permission lets a failed multipart upload clean
    up after itself. Miru never needs read, list, or delete permissions on your data.

    Console: **IAM → Roles → miru-uploader → Add permissions → Create inline policy →
    JSON**, and paste:

    ```json
    {
      "Version": "2012-10-17",
      "Statement": [
        {
          "Effect": "Allow",
          "Action": [
            "s3:PutObject",
            "s3:AbortMultipartUpload"
          ],
          "Resource": "arn:aws:s3:::<bucket-name>/*"
        }
      ]
    }
    ```

    <Note>
      [Verification](/data-uploads/buckets/manage#verify-a-bucket) writes a
      zero-byte probe object under `.miru/probe/` and then attempts to delete it.
      Cleanup is best-effort — verification succeeds without it — but if you want the
      probe objects removed, also grant `s3:DeleteObject` on
      `arn:aws:s3:::<bucket-name>/.miru/probe/*`.
    </Note>

    {/* TODO (product): confirm whether SSE-KMS buckets need an additional kms:GenerateDataKey grant for the role. */}

    CLI: save the JSON above as `put-policy.json`, then:

    ```bash
    aws iam put-role-policy \
      --role-name miru-uploader \
      --policy-name miru-put-object \
      --policy-document file://put-policy.json
    ```

    <Frame>
      ![IAM role inline policy editor showing the s3:PutObject and s3:AbortMultipartUpload permissions scoped to the bucket](https://assets.mirurobotics.com/docs/v04/images/data-uploads/buckets/aws/put-object-policy.png)
    </Frame>
    {/* TODO (screenshot): AWS Console, IAM → role → Add permissions → Create inline policy → JSON editor, highlight the s3:PutObject and s3:AbortMultipartUpload actions and the bucket Resource ARN. */}
  </Step>

  <Step title="Register the bucket in Miru">
    From the **Buckets** page in Miru, select **New bucket**, choose **AWS**, and
    supply:

    - `name` — the bucket from step 1
    - `region` — the bucket's region
    - `role_arn` — the role ARN from step 2

    Select **Create**. Miru issues the bucket's **external ID** and returns it —
    copy it for the next step.

    {/* TODO (product): confirm where registration surfaces — dashboard or CLI — and where the issued external ID is displayed. */}

    <Frame>
      ![Miru New bucket form for the AWS provider showing the name, region, and role ARN fields and the issued external ID](https://assets.mirurobotics.com/docs/v04/images/data-uploads/buckets/aws/register-bucket.png)
    </Frame>
    {/* TODO (screenshot): Miru dashboard, Buckets → New bucket form with provider set to AWS, highlight the name, region, and role_arn fields and the external_id Miru returns after creation. */}
  </Step>

  <Step title="Add the external ID to the trust policy">
    Update the role's trust policy to **require the external ID Miru issued** via an
    `sts:ExternalId` condition. The external ID is the guard against the
    [confused-deputy](https://docs.aws.amazon.com/IAM/latest/UserGuide/confused-deputy.html)
    problem: even another tenant of the same Miru service cannot trick Miru into
    assuming your role without presenting your specific external ID.

    Console: **IAM → Roles → miru-uploader → Trust relationships → Edit trust
    policy**, and replace with:

    ```json
    {
      "Version": "2012-10-17",
      "Statement": [
        {
          "Effect": "Allow",
          "Principal": { "AWS": "<miru-integration-role-arn>" },
          "Action": "sts:AssumeRole",
          "Condition": {
            "StringEquals": { "sts:ExternalId": "<external-id>" }
          }
        }
      ]
    }
    ```

    CLI: save the JSON above as `trust-policy.json`, then:

    ```bash
    aws iam update-assume-role-policy \
      --role-name miru-uploader \
      --policy-document file://trust-policy.json
    ```

    <Frame>
      ![IAM trust policy editor showing the sts:ExternalId condition added to the Miru principal](https://assets.mirurobotics.com/docs/v04/images/data-uploads/buckets/aws/external-id-condition.png)
    </Frame>
    {/* TODO (screenshot): AWS Console, IAM → role → Trust relationships → Edit trust policy, highlight the sts:ExternalId condition value. */}
  </Step>

  <Step title="Verify the bucket">
    Back in Miru, run **Verify** on the bucket to exercise the full AssumeRole chain
    end to end. The bucket's [status](/data-uploads/buckets/overview#properties) flips
    to `verified` on success, or `invalid` with the failure reason in
    `verification_error`.

    <Frame>
      ![Miru bucket page showing the Verify action and a verified status](https://assets.mirurobotics.com/docs/v04/images/data-uploads/buckets/aws/verify-bucket.png)
    </Frame>
    {/* TODO (screenshot): Miru dashboard, bucket detail/menu with the Verify action and the resulting verified status badge. */}
  </Step>
</Steps>

### How device credentials are minted at runtime

When a device requests an upload, the Miru control plane assumes your `role_arn` via
**STS AssumeRole** (presenting the matching `external_id`), and downscopes the
resulting session to a **single object key** — one carrying the requesting device's
own verified ID. The device receives those short-lived session credentials and drives
a native **S3 multipart upload** directly to your bucket; if the credentials expire
mid-upload, it requests fresh credentials for the same key and resumes.

The control plane never receives a long-lived key, and the device never touches your
AWS account credentials — only a short-lived session scoped to creating one object.
