A GCS bucket is authenticated via **Workload Identity Federation (WIF)** — the
native mechanism Google Cloud provides for letting an external identity act in your
project *without* a service-account key. See [Connecting your bucket](#connecting-your-bucket)
for how these values are produced.

## Properties

<ParamField path="project_id" type="string" required>
  <ImmutableBadge />

  The GCP project ID that owns the bucket and the Workload Identity Federation
  configuration.
</ParamField>

<ParamField path="bucket_name" type="string" required>
  <ImmutableBadge />

  The name of the GCS bucket that uploads are written to.
</ParamField>

<ParamField path="wip_provider" type="string" required>
  <ImmutableBadge />

  The full Workload Identity Pool provider resource name used to federate Miru's
  identity into your GCP project.

  Example: `projects/123456789/locations/global/workloadIdentityPools/miru-pool/providers/miru-provider`
</ParamField>

<ParamField path="service_account_email" type="string" required>
  <ImmutableBadge />

  The email of the GCP service account Miru impersonates to mint V4-signed upload
  URLs.

  Example: `miru-uploader@my-gcp-project.iam.gserviceaccount.com`
</ParamField>

## Connecting your bucket

No secrets change hands in either direction. The integration is established by
exchanging **identifiers and configuration only**:

| Miru provides to you | You provide to Miru |
| --- | --- |
| Miru's AWS account ID | Your GCP project ID |
| Miru's IAM role ARN | Workload Identity Pool + provider resource path |
| | The signing service account email |
| | The destination bucket name |

There are **no service-account keys, passwords, or long-lived tokens** in this list —
by design. Nothing you hand Miru can be replayed or exfiltrated to reach your data.

### How a credential is minted at runtime

When a device requests an upload, the Miru control plane performs the following
exchange with Google Cloud. The device is uninvolved here.

<img
  src="https://assets.mirurobotics.com/docs/v04/images/data-uploads/credential-mint.svg"
  alt="Credential minting: the Miru control plane assumes its AWS role, presents that identity to Google STS (which checks it against your Workload Identity pool's attribute condition) for a short-lived federated token, then acts as your signing service account to mint a V4 signed URL. The chain is keyless — no service-account key is ever downloaded."
/>

1. The Miru control plane assumes its **AWS role**.
2. It presents that identity to **Google STS**, which checks it against your pool's
   attribute condition and returns a **short-lived federated Google token**.
3. Using that token, the control plane acts as your signing service account to
   produce a **V4 signed URL**.
4. The device uses the signed URL to perform the upload to your bucket.

The chain is keyless end to end — no service-account key is ever downloaded.
