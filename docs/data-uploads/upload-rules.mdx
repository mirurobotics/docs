A rule has two parts — a `source` (what to collect on the device) and a `destination`
(where it goes in your bucket):

```yaml
source:
  glob: "/var/log/robot/*.log"        # absolute glob (must start with /)
  poll_interval: 60s                  # how often the agent checks for eligible files
  stability_window: 60s               # quiescence before a file is considered finished

destination:
  bucket_id: bkt_123                  # a registered bucket
  path: "logs/{device_id}/{filename}" # object-path template; must include {device_id}
  delete_policy: never                # never | after_upload
```

## Properties

<ParamField path="upload_type" type="Upload Type">
  The [upload type](/data-uploads/upload-types) this rule belongs to.
</ParamField>

<ParamField path="source" type="Source">
  Where data is collected on the device. See [Source](#source).
</ParamField>

<ParamField path="destination" type="Destination">
  Where collected data is written, and how the local copy is handled. See
  [Destination](#destination).
</ParamField>

## Source

<ParamField path="glob" type="string" required>
  An absolute glob pattern (must start with `/`) that selects the files to upload.

  Example: `/var/log/robot/*.log`
</ParamField>

<ParamField path="poll_interval" type="duration" default="60s">
  How often the agent checks matching files for upload eligibility.
</ParamField>

<ParamField path="stability_window" type="duration" default="60s">
  How long a matching file's size and modification time must stay unchanged
  (quiescent) before it is considered finished and eligible for upload. Files in a
  format with a finalization marker (such as MCAP or parquet) are detected directly;
  this window is the fallback for other files. See
  [completion detection](/data-uploads/uploads#completion-detection).
</ParamField>

## Destination

<ParamField path="bucket_id" type="string" required>
  ID of a registered [bucket](/data-uploads/buckets) (`bkt_…`) this rule uploads to.
</ParamField>

<ParamField path="path" type="string" required>
  The object-path template the collected file is written to. **Must contain
  `{device_id}`** so each device's uploads are namespaced under its own verified ID —
  this is what guarantees
  [device data isolation](/data-uploads/buckets#device-data-isolation). Where the
  token sits and how you lay out the rest of the path is up to you.
</ParamField>

<ParamField path="delete_policy" type="enum" default="never">
  When the device deletes the local source file after collecting it.

  - `never` *(default)* — keep the local file in place; you manage local retention
    yourself.
  - `after_upload` — delete the local file once the upload is durably confirmed.
</ParamField>

## Immutability and releases

Upload rules are **immutable** and deduplicated by digest — defining an identical rule
reuses the existing one rather than creating a duplicate. A rule is created once, then
included in a [release](/learn/releases/overview) by ID and shipped to devices as part
of that release.

Defining what data leaves your devices is therefore the same kind of versioned,
reviewable, staged operation as defining anything else you ship.

## Create an upload rule  {/* TODO: confirm required role/scope */}

From the **Upload Rules** area, select **New upload rule**. Pick the
[upload type](/data-uploads/upload-types) it belongs to, define its [`source`](#source)
(the glob and completion settings) and its [`destination`](#destination) (the target
`bucket_id`, the `path` template, and the `delete_policy`), then create it.

Because rules are deduplicated by digest, creating a rule identical to an existing one
returns that existing rule rather than a duplicate. Once created, attach the rule to a
[release](/learn/releases/overview) by its ID to ship it to devices.

## Editing and deleting rules  {/* TODO: confirm required role/scope */}

Upload rules **cannot be edited or deleted individually** — they are immutable by
design. To change what a rule collects or where it goes, create a **new** rule with the
updated definition and ship it in a new [release](/learn/releases/overview). The
release a device is on determines which rules are active, so retiring a rule is a matter
of shipping a release that no longer includes it.
