[Upload rules](/data-uploads/upload-rules/overview) reach your devices the same way
config schemas do — as part of a [release](/fundamentals/releases/overview) created
via the CLI. You define each rule as a **YAML file** living in the same Git repository
as your code, and `miru release create` pushes the rule files alongside your schemas.
Defining what data leaves your fleet is versioned, reviewable, and traceable to a
commit, and the CLI can run in a [CI pipeline](/developers/ci/overview) just like it
does for [config schemas](/cfg-mgmt/releasing-config-schemas).

## Prerequisites

To release upload rules, the following prerequisites must be met:

1. CLI installed - [install](/developers/cli/install) the Miru CLI on your development
   machine and authenticate using the [login](/developers/cli/login) command
1. Local Git repository - the rule files must be committed to a local Git repository
   (the CLI refuses dirty or untracked rule files)
1. Registered bucket - each rule references a
   [bucket](/data-uploads/buckets/overview) by name, so the bucket must be registered
   first
1. At least one config schema - a release must contain at least one
   [config schema](/cfg-mgmt/releasing-config-schemas); upload rules ride along with
   it

## Upload rule files

Each upload rule is one YAML file:

```yaml robot-logs.yaml
collection_slug: robot-logs

source:
  glob: "/var/log/robot/*.log"
  stability_window_secs: 60

destination:
  bucket: my-uploads-bucket
  path: "logs/{device_id}/{upload_id}/{file_name}"
  delete_policy: never
```

<ParamField path="collection_slug" type="string" required>
  The slug of the [upload collection](/data-uploads/upload-collections) the rule
  belongs to. If no collection with this slug exists yet, the CLI creates it for you.

  Each rule file in a release must reference a **distinct** collection — the CLI
  rejects two files sharing a `collection_slug`.
</ParamField>

<ParamField path="source.glob" type="string" required>
  An absolute glob pattern (must start with `/`) selecting the files to upload. See
  [Sources](/data-uploads/upload-rules/sources).
</ParamField>

<ParamField path="source.stability_window_secs" type="integer" default="60">
  How long, in seconds, a matching file must stay unchanged before it is considered
  finished. See [Sources](/data-uploads/upload-rules/sources).
</ParamField>

<ParamField path="destination.bucket" type="string" required>
  The **name** of the registered [bucket](/data-uploads/buckets/overview) the rule
  uploads to.
</ParamField>

<ParamField path="destination.path" type="string">
  The object-path template. Optional — omit it to use the server default. When
  provided, it must contain `{device_id}` and `{upload_id}`. See
  [Destinations](/data-uploads/upload-rules/destinations).
</ParamField>

<ParamField path="destination.delete_policy" type="enum" default="never">
  When the device deletes the local source file after collecting it: `never` or
  `after_upload`. See [Destinations](/data-uploads/upload-rules/destinations).
</ParamField>

Rules are identified by the digest of their resolved definition. Releasing a rule file
identical to an existing rule in the same collection attaches that **existing** rule
to the release rather than creating a duplicate — so re-releasing unchanged rule files
is always safe.

## Git commit

The CLI captures the same Git metadata for upload rules as it does for
[config schemas](/cfg-mgmt/releasing-config-schemas#git-commit): the commit SHA, the
repository's origin URL, and each rule file's path relative to the repository root.
Every rule permanently records the commits it was released from, so any rule in Miru
traces back to the exact lines of YAML that defined it — see
[Git commit metadata](/data-uploads/upload-rules/manage#git-commit-metadata).

Git metadata is not optional: rule files must live in a local Git repository with a
remote URL, and their latest changes must be committed (not dirty or unstaged), or the
CLI fails to create the release.

## CLI command

Upload rules are passed to `miru release create` with two flags, alongside the usual
`--version` and schema flags:

<ParamField path="--upload-rule" type="string">
  The path to an upload rule file to include in the release. May be specified multiple
  times to include multiple rules.

  Examples: `./upload-rules/robot-logs.yaml`
</ParamField>

<ParamField path="--upload-rules" type="string">
  A directory containing upload rule files to include in the release. May be specified
  multiple times for multiple directories. Only YAML files in the directory are picked
  up.

  Examples: `./upload-rules/`
</ParamField>

### Examples

<Tabs>
  <Tab title="Rule files">
    Use the `--upload-rule` flag to specify individual rule files.

    ```bash
    miru release create \
      --version v1.0.0 \
      --schema ./schemas/mobility.json \
      --upload-rule ./upload-rules/robot-logs.yaml \
      --upload-rule ./upload-rules/crash-reports.yaml
    ```
  </Tab>
  <Tab title="Rule directory">
    Use the `--upload-rules` flag to specify a directory of rule files.

    ```bash
    miru release create \
      --version v1.0.0 \
      --schemas ./schemas/ \
      --upload-rules ./upload-rules/
    ```
  </Tab>
</Tabs>

A single release carries both the upload rules and the
[config schemas](/cfg-mgmt/releasing-config-schemas), so one versioned, staged change
defines what your fleet runs and what it collects.

## Shared release mechanics

Versioning, the staging area, Git-metadata capture, and the full set of
`release create` flags are common to every release, regardless of what it contains.
See the [Releases](/fundamentals/releases/overview) overview and
[Releasing config schemas](/cfg-mgmt/releasing-config-schemas) for the shared
workflow, or the [CLI reference](/references/cli/release-create) for a concise command
summary.
