A rule has two parts — a `source` (what to collect on the device) and a `destination`
(where it goes in your bucket) — and belongs to an
[upload collection](/data-uploads/upload-collections) that gives its data stream a
stable identity. As a [rule file](/data-uploads/defining-releases):

```yaml
collection_slug: robot-logs             # the collection this rule belongs to

source:
  glob: "/var/log/robot/*.log"          # absolute glob (must start with /)
  stability_window_secs: 60             # quiescence before a file is considered finished

destination:
  bucket: my-uploads-bucket             # a registered bucket, referenced by name
  path: "logs/{device_id}/{upload_id}/{file_name}"  # object-path template
  delete_policy: never                  # never | after_upload
```

## Properties

<ParamField path="upload_collection_id" type="string">
  <ImmutableBadge />

  ID of the [upload collection](/data-uploads/upload-collections) this rule belongs to
  (`upl_col_…`). In rule files, the collection is referenced by its `collection_slug`
  instead.
</ParamField>

<ParamField path="digest" type="string">
  <ImmutableBadge />

  A hash of the rule's resolved definition — its collection, source, and destination.
  Upload rules are deduplicated by digest within their collection: creating a rule
  identical to an existing one returns the existing rule instead of a duplicate.

  Example: `sha256:1234567890`
</ParamField>

<ParamField path="source" type="Source">
  <ImmutableBadge />

  What to collect on the device: the file glob and the completion settings. See
  [Sources](/data-uploads/upload-rules/sources).
</ParamField>

<ParamField path="destination" type="Destination">
  <ImmutableBadge />

  Where collected files are written — the target bucket and object-path template —
  and how the local copy is handled. See
  [Destinations](/data-uploads/upload-rules/destinations).
</ParamField>

## Immutability and releases

Upload rules are **immutable**. A rule is created once, shipped to devices as part of
a [release](/fundamentals/releases/overview), and never edited in place — to change
what is collected, you release a revised rule and the old one simply stops being
referenced. Defining what data leaves your devices is therefore the same kind of
versioned, reviewable, staged operation as defining anything else you ship.

Because rules are deduplicated by digest within their collection, releasing the same
rule definition twice never creates a duplicate — the release simply references the
existing rule.

## Git provenance

Rules are defined as YAML files in your Git repository and created by
[releasing them via the CLI](/data-uploads/defining-releases). Each rule records
the Git commits it was released from — the commit and the rule file's path within the
repository — so any rule in Miru can be traced back to the exact lines of YAML that
defined it. See [Manage rules](/data-uploads/upload-rules/manage#git-commit-metadata).
